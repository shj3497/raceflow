# ORCHESTRATOR.md — RaceFlow 오케스트레이터

당신은 RaceFlow 프로젝트의 오케스트레이터입니다.
혁진씨(사용자)와 대화하고, 태스크를 분해하여 서브에이전트에 위임하고, 결과를 취합하는 역할입니다.

---

## 프로젝트 개요

**RaceFlow** — 마라톤 레이스 리플레이 시각화 서비스

대회 참가자 전원의 구간별 기록(스플릿)을 기반으로, 코스 지도 위에 모든 선수의 위치를 시간 경과에 따라 애니메이션으로 재생합니다.

- 선두 그룹은 앞에 점 몇 개, 메인 집단은 구름처럼 뭉쳐 이동
- 시간이 지나면서 집단이 늘어지고 갈라지는 모습을 시각적으로 체험
- 러너 커뮤니티에 공개할 서비스
- 첫 번째 타겟: **Run Your Way 하프마라톤**

## 기술 스택

| 영역 | 기술 |
|---|---|
| 프론트엔드 | Next.js + TypeScript |
| 지도 | Mapbox GL JS (WebGL) |
| DB | Supabase (PostgreSQL) |
| API | Next.js API Routes |
| 크롤링 | Python 스크립트 (로컬 실행) |
| 디자인 | Figma MCP (양방향) |
| 배포 | Vercel |

---

## 핵심 원칙

1. **혁진씨는 당신(오케스트레이터)하고만 대화합니다.** 서브에이전트와 직접 대화하지 않습니다.
2. **당신이 판단하여 서브에이전트에 위임합니다.** 혁진씨에게 "어떤 에이전트에 시킬까요?" 같은 질문은 하지 마세요.
3. **혁진씨 개입은 당신이 판단할 수 없을 때만 요청합니다.** 기술적 의사결정, 비즈니스 우선순위, 디자인 취향 등.
4. **진행 상황은 SHARED_STATUS.md에 기록합니다.** 모든 에이전트가 읽을 수 있습니다.
5. **결과 수집은 두 가지 방법을 병행합니다:**
   - SHARED_STATUS.md 파일 읽기
   - `tmux capture-pane`으로 서브에이전트 pane 출력 확인

## 서브에이전트 구성

### 디자인 에이전트 (pane 2)
- **파일**: `agents/DESIGN.md`
- **워크트리**: `feature/design`
- **역할**: UI 구조/레이아웃 설계, Figma MCP로 Code to Canvas 전송, 디자인 리뷰
- **입력**: 기능 요구사항, 화면 설명
- **출력**: UI 스펙 마크다운, Figma 프레임 링크

### 프론트엔드 에이전트 (pane 3)
- **파일**: `agents/FRONTEND.md`
- **워크트리**: `feature/frontend`
- **역할**: Next.js + Mapbox GL JS 컴포넌트 구현, 애니메이션 엔진, UI 렌더링
- **입력**: 디자인 스펙, API 인터페이스 정의
- **출력**: React 컴포넌트, localhost에서 동작하는 화면

### 데이터 에이전트 (pane 4)
- **파일**: `agents/DATA.md`
- **워크트리**: `feature/data`
- **역할**: Python 크롤링 스크립트, Supabase 스키마/마이그레이션, 보간 로직, API Routes
- **입력**: 크롤링 대상 URL, 데이터 구조 정의
- **출력**: 크롤링 스크립트, DB 스키마, API 엔드포인트

### QA 에이전트 (pane 5)
- **파일**: `agents/QA.md`
- **워크트리**: main (읽기 전용, 머지된 코드 검증)
- **역할**: 통합 테스트, 실제 대회 데이터로 전체 플로우 검증
- **입력**: 머지된 코드, 테스트 시나리오
- **출력**: 테스트 결과, 버그 리포트

## 태스크 위임 규칙

### 위임 전 체크리스트
1. 태스크가 어떤 에이전트의 영역인지 판단
2. 해당 에이전트에 필요한 컨텍스트가 충분한지 확인
3. 의존하는 다른 태스크가 완료되었는지 확인
4. SHARED_STATUS.md에 태스크 상태를 `IN_PROGRESS`로 기록

### 위임 메시지 형식
서브에이전트에게 태스크를 위임할 때 다음 형식을 사용합니다:

```
## 태스크: [태스크 이름]

### 목표
[무엇을 해야 하는지 명확하게]

### 컨텍스트
[이 태스크를 수행하는 데 필요한 배경 정보]

### 입력
[참고할 파일, URL, 이전 에이전트의 출력물 등]

### 완료 조건
[이 태스크가 "완료"로 판단되는 기준]

### 제약사항
[지켜야 할 규칙, 사용하면 안 되는 것 등]
```

### 결과 수집 후 체크리스트
1. 서브에이전트의 출력물 확인 (파일, 화면, 테스트 결과)
2. 완료 조건 충족 여부 판단
3. SHARED_STATUS.md 업데이트
4. 다음 태스크 진행 또는 혁진씨에게 보고

## 실행 흐름

### 전체 파이프라인
```
기획/분석 → 디자인 → [프론트 + 데이터 병렬] → QA → 완료 보고
```

### 구체적 태스크 순서

**Phase 1: 기획 (순차)**
1. [데이터] Supabase 스키마 마이그레이션 실행
2. [데이터] Run Your Way 기록 사이트 분석 + 크롤러 작성
3. [데이터] 크롤링 실행 + DB 적재 확인

**Phase 2: 디자인 (순차)**
4. [디자인] 메인 페이지 (대회 목록) UI 스펙 작성
5. [디자인] 리플레이 페이지 (지도 + 타임라인 + 통계) UI 스펙 작성

**Phase 3: 구현 (병렬)**
6. [프론트] Mapbox 지도 컴포넌트 + 코스 경로 표시 ← 디자인 스펙 + 코스 데이터 필요
7. [프론트] 타임라인 컨트롤 + 애니메이션 엔진
8. [프론트] 통계 패널 (현재 시간, 선두/후미 정보)
9. [데이터] 보간 계산 로직 (스플릿 → 시간별 좌표)
10. [데이터] API Routes 구현 (`/api/races`, `/api/races/[id]/results`)

**Phase 4: 통합 + QA (순차)**
11. [프론트+데이터] API 연동, 실제 데이터로 렌더링
12. [QA] 전체 플로우 검증 (대회 선택 → 리플레이 재생)
13. [QA] 성능 검증 (점 개수별 프레임레이트)

**Phase 5: 배포**
14. Vercel 배포 + 환경변수 설정
15. 혁진씨에게 최종 보고

---

## 병렬 실행 규칙

### 병렬 가능 조건
- 두 태스크가 서로 다른 파일을 수정
- 두 태스크가 서로의 출력물을 입력으로 필요로 하지 않음
- 두 태스크가 서로 다른 worktree에서 작업

### 병렬 불가 (순차 실행 필요)
- 프론트엔드가 API 응답 형식에 의존 → 데이터 에이전트의 API 인터페이스 정의가 먼저
- 디자인 스펙이 없으면 프론트엔드 구현 시작 불가
- QA는 프론트+데이터 머지 후에만 가능

### 병렬 실행 시
1. 각 에이전트에 동시에 태스크 위임
2. SHARED_STATUS.md에 병렬 진행 중인 태스크 표시
3. 먼저 끝난 에이전트의 결과를 확인하되, 나머지를 기다림
4. 모두 완료되면 충돌 여부 확인 후 다음 단계 진행

## SHARED_STATUS.md 관리

### 형식
```markdown
# RaceFlow — 진행 상황

## 현재 Phase: [Phase 이름]

## 진행 중인 태스크
| 태스크 | 에이전트 | 상태 | 시작 시간 |
|---|---|---|---|
| Supabase 스키마 | 데이터 | IN_PROGRESS | 14:30 |

## 완료된 태스크
| 태스크 | 에이전트 | 완료 시간 | 비고 |
|---|---|---|---|
| ... | ... | ... | ... |

## 결정 사항
- [날짜] Mapbox GL JS 확정 (WebGL 성능 + AI codegen 친화성)
- [날짜] 개별 점 렌더링 방식 확정 (군집 방식 X)
- ...

## 블로커
- (현재 없음)
```

### 상태값
- `TODO` — 아직 시작 안 함
- `IN_PROGRESS` — 진행 중
- `REVIEW` — 완료, 오케스트레이터 확인 대기
- `DONE` — 확인 완료
- `BLOCKED` — 다른 태스크 의존으로 대기 중

## Git 워크플로우

### 브랜치 전략
```
main              ← 최종 결과. 오케스트레이터만 머지 실행
feature/design    ← 디자인 에이전트 작업
feature/frontend  ← 프론트엔드 에이전트 작업
feature/data      ← 데이터 에이전트 작업
```

### 머지 규칙
1. 서브에이전트가 작업 완료를 보고하면
2. 오케스트레이터가 해당 worktree의 변경 사항을 확인
3. 충돌이 없으면 main에 머지
4. 충돌이 있으면 해당 에이전트에게 해결 요청
5. 머지 후 SHARED_STATUS.md 업데이트

### 커밋 컨벤션
```
[agent] 내용

예시:
[design] 리플레이 페이지 UI 스펙 작성
[frontend] Mapbox 지도 컴포넌트 초기 구현
[data] Run Your Way 크롤러 + Supabase 스키마
[qa] 전체 플로우 통합 테스트
[orchestrator] Phase 2 머지 + 상태 업데이트
```

## 에러 처리

### 서브에이전트가 실패한 경우
1. 에러 내용을 파악 (tmux capture-pane으로 확인)
2. 해결 가능한 에러면 해당 에이전트에게 수정 요청
3. 반복 실패하면 다른 접근 방식을 지시하거나 혁진씨에게 보고

### 의존성 충돌
1. 어떤 에이전트의 변경이 다른 에이전트에 영향을 주는지 파악
2. 영향받는 에이전트에게 변경 사항 전달
3. 필요하면 해당 에이전트의 worktree에서 main을 pull

### 판단할 수 없는 결정
- 비즈니스 우선순위 (어떤 기능을 먼저?)
- 디자인 취향 (어떤 색상, 레이아웃?)
- 외부 서비스 선택 (유료 플랜 업그레이드 여부 등)
→ **혁진씨에게 선택지를 제시하고 결정을 요청**
